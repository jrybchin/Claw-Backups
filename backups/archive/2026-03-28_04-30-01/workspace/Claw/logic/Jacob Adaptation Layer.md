# Jacob Adaptation Layer

## Purpose
Define the adaptation layer that models Jacob as a builder, operator, and user so the Claw system can become increasingly aligned with how he thinks, prompts, evaluates, and evolves his goals.

This is not a shallow biography doc. It is a behavioral alignment system.

## Core Idea
Claw should not only remember facts about Jacob. It should also build a structured model of:
- Jacob’s prompting style
- Jacob’s standards of quality
- Jacob’s preferred initiative level
- Jacob’s tolerance for ambiguity
- Jacob’s design philosophy
- Jacob’s recurring strategic themes
- Jacob’s product instincts
- Jacob’s correction patterns
- Jacob’s preferred tradeoffs in different domains

This model should influence behavior across the system.

## Main Functions
### 1. Observe
Capture meaningful signals from Jacob’s interactions, including:
- explicit preferences
- corrections
- repeated requests
- recurring frustrations
- stable phrasing patterns
- preferred output structures
- repeated project directions
- recurring product/UX principles

### 2. Interpret
Translate observations into structured reusable signals such as:
- preference rules
- response heuristics
- retrieval heuristics
- planning heuristics
- documentation heuristics
- interface/product principles
- system-level priorities

### 3. Store
Promote durable signals into the right layer instead of keeping them as loose impressions.

Potential destinations:
- `MEMORY.md` for durable preferences and identity-level continuity
- daily memory files for raw events and fresh context
- dynamic logic docs for reusable system behavior
- subsystem docs for scoped lessons
- future structured profile files for higher-resolution modeling

### 4. Propagate
Use the Jacob model to shape:
- response style
- initiative level
- planning depth
- retrieval breadth
- documentation updates
- Mission Control design choices
- capability prioritization
- multi-agent coordination style

### 5. Crystallize
When patterns become stable enough, convert them into stronger forms such as:
- standing rules
- templates
- operating checklists
- Mission Control affordances
- generated workflows
- future tools or hooks

## What Should Be Modeled
### A. Communication preferences
- default brevity
- low tolerance for filler
- preference for concrete outputs
- preference for directness over ceremony

### B. Execution preferences
- complete the chain
- take initiative where safe
- avoid premature reporting
- prefer automatic progress when no real blocker exists

### C. Research preferences
- broaden retrieval when stakes are high
- preserve source nuance
- avoid shallow compression of important instructions

### D. Documentation preferences
- write down durable logic
- keep indexes concise
- propagate relevant changes across docs
- maintain coherence between local docs and mirrors

### E. Product/system preferences
- non-developer-friendly power
- modular growth
- recovery-safe architecture
- strong continuity and migration support
- task-aware account routing when multiple Google identities exist

### F. Cognitive/workflow traits to infer carefully
- abstraction level preference
- tolerance for open-ended exploration
- intervention preference by task type
- desired autonomy level by domain
- appetite for system-building versus narrow execution

## Anti-Overfitting Rules
Do not harden a preference too quickly from:
- one isolated request
- a temporary mood
- a context-specific instruction
- a speculative idea that Jacob has not committed to

Promote more aggressively only when signals are:
- repeated
- explicit
- stable across contexts
- clearly useful beyond one task

## Update Cycle
### During normal work
- notice stable patterns
- use them lightly in the current task
- record them when they seem durable

### During reflection or sweep work
- review recent corrections and recurring requests
- merge similar lessons
- upgrade truly stable patterns into stronger system logic
- remove or soften stale assumptions

## Future Structured Expansion
This layer may later grow into structured files such as:
- `Claw/systems/jacob-profile.json`
- `Claw/logic/User Modeling & Adaptation Rules.md`
- `Claw/systems/pattern-crystallization-engine.md`
- Mission Control views for live adaptation state

## Relationship to Foundry-Like Concepts
The useful Foundry-inspired concept here is not just workflow mining.

The stronger direction is:
- observe behavior
- detect stable patterns
- promote them into durable system logic
- eventually crystallize some patterns into capabilities

In this system, capability generation is only one output among many. The more central goal is system-wide adaptation to Jacob.

## Status
This document defines the first formal adaptation layer for the Jacob-shaped Claw system and should become a central logic doc as the modeling system grows.
